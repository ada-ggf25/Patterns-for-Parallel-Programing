# A1 — Numerical Integration (20 pts)

> **Active assessment.** See [[A1 Progress]] for the step-by-step checklist and current status.

Parallelise a composite-trapezoid integration kernel with **non-uniform per-iteration cost**, then analyse schedule performance across the canonical thread ladder.

## Kernel

Integrate `f(x)` over `[0, 1]` with N = 100,000,000 subintervals using the **composite-trapezoid rule**. `f(x)` has a deliberate **spike region** for `x ∈ (0.3, 0.4)` running ~10× slower than elsewhere — schedule choice is genuinely informative.

The actual serial loop in `integrate_serial.cpp` (do not modify):

```cpp
const double h = (b - a) / static_cast<double>(n);
double sum = 0.5 * (f(a) + f(b));   // endpoints get half weight — outside the parallel loop
for (long i = 1; i < n; ++i) {
    const double x = a + (static_cast<double>(i) * h);
    sum += f(x);
}
return sum * h;
```

The parallel version in `integrate.cpp` must wrap only the inner loop. The half-weight endpoint terms `0.5*(f(a)+f(b))` are computed once outside and must **not** be included in the parallel reduction. Template:

```cpp
double sum = 0.5 * (f(a) + f(b));
#pragma omp parallel for default(none) shared(a, h, n) reduction(+:sum) schedule(???)
for (long i = 1; i < n; ++i) {
    const double x = a + (static_cast<double>(i) * h);
    sum += f(x);
}
```

### Why the spike creates load imbalance

With `schedule(static)`, each thread gets a contiguous chunk of iterations. The spike iterations (i ≈ 0.3N to 0.4N) all land on one or two threads. Those threads take ~10× longer than the others; the fast threads finish and idle at the implicit barrier:

```
Thread 0:  ████ DONE ——————————————— wait
Thread 1:  ████████████████████████ (has the spike range)
Thread 2:  ████ DONE ———————————————
...
Thread 127:████ DONE ———————————————
            ↑ all idle until thread 1 finishes
```

`dynamic` or `guided` distribute work on demand so expensive iterations spread across the team.

### Chunk size tuning (`dynamic, C`)

| C value | Problem |
|---|---|
| C = 1 | Queue-dispatch overhead per chunk dominates at high thread counts |
| C = N/P | Same as static — spikes cluster on one thread |
| C = 64 | Good starting point for spike-every-10% workload |

For A1, start with `C ∈ {32, 64, 128}` and pick the timing winner at each thread count. Rule of thumb: chunk should contain ~5–10 spike-spacings worth of iterations so dispatch overhead < 5 % of chunk compute time.

### Schedule winners by thread count (expected)

| Thread count | Typical winner | Why |
|---|---|---|
| 1 | All tie | Single thread; no scheduling at all |
| 16 | `dynamic` or `guided` | Spike is 10 % of domain; static gives one thread all of them |
| 64 | `dynamic, 64` | More threads → imbalance worse; dynamic distributes best |
| 128 | `dynamic, 64` | Amplified imbalance; static efficiency drops to ~10–15 % |

## REFLECTION Guide

Each section requires ≥ 50 words and must be grounded in your measured numbers.

### Section 1 — Schedule choice and why

1. Name the winner (e.g., `schedule(dynamic, 64)`).
2. Explain the cost structure of f(x) — spike region means non-uniform per-iteration cost.
3. State what static gave you: times at each thread count; name the schedule you tried and discarded.
4. Explain why dynamic/guided wins: load rebalancing distributes the spike across all threads.

> Example framing: "Static gave 2.4 s at 128 threads while dynamic (chunk=64) gave 0.6 s — a 4× gap. The spike region clusters in the range i ≈ 0.3N–0.4N; with static these fall on threads 38–51, creating a ~10× imbalance. Dynamic eliminates this because threads pull the next chunk immediately on completion."

### Section 2 — Scaling behaviour

1. Identify the breakpoint: e.g., "speedup is near-linear up to 16T, then flattens."
2. Attribute to a cause: Amdahl's serial fraction, load imbalance (if static), NUMA effects at 64T+.
3. Report efficiency values from your `tables.csv` — e.g., "efficiency drops from 0.92 at 16T to 0.61 at 128T."
4. Expected efficiency ranges with a well-parallelised dynamic schedule:
   - 16T: ~85–95 % (very good)
   - 64T: ~70–85 % (NUMA starts to matter)
   - 128T: ~55–75 % (cross-socket NUMA penalty)

Departure from linear speedup comes from: fork-join overhead, NUMA cost (two sockets at 128T), memory bandwidth saturation, and reduction overhead (log₂128 = 7 tree steps).

### Section 3 — Roofline position

A1 integration is **compute-bound** (OI >> ridge point). Rome roofline constants (from the day-2 slides — use these exact values in the REFLECTION):

| Ceiling | Value |
|---|---|
| Theoretical peak DP | 4608 GFLOPs |
| HPL-achievable | 2896 GFLOPs |
| STREAM Triad (canonical) | **246 GB/s** |
| Ridge OI | 4608 / 246 ≈ **18.7 FLOPs/byte** |

> The assignment brief specifies 246 GB/s (the 32T per-CCX STREAM value) as the canonical ceiling. Use this number — not 231.5 GB/s — for the ridge point in your REFLECTION.

### FLOPs breakdown for `f(x)`

Count the operations executed per call to estimate OI and achieved GFLOPs:

**Base case** (x outside spike, ~90 % of iterations):
```
std::sin(x)       ≈ 1 transcendental  (treated as 1 FLOP for counting)
std::cos(x)       ≈ 1 transcendental
sin * cos         = 1 multiply
x * x             = 1 multiply
y + (x*x)         = 1 add
                  ≈ 5 FLOPs total
```

**Spike case** (x ∈ (0.3, 0.4), ~10 % of iterations):
```
sin, cos, mul     = 5 FLOPs (same as base)
10 × loop body:
  std::abs(y)     = 1 FLOP each → 10
  y + 1.0         = 1 add each  → 10
  std::sqrt(...)  = 1 sqrt each → 10
                  ≈ 35 FLOPs total
```

**Weighted average per iteration:**
```
weighted_avg = 0.9 × 5 + 0.1 × 35 = 4.5 + 3.5 = 8 FLOPs/iter
total_FLOPs  = 8 × N  =  8 × 1e8  =  8×10^8  = 0.8 GFLOPs
achieved_GFLOPs = 0.8 / T(P)          (T in seconds)
fraction_vs_peak = achieved_GFLOPs / 4608
fraction_vs_HPL  = achieved_GFLOPs / 2896
```

Example: if T(128) = 0.05 s → achieved = 0.8/0.05 = 16 GFLOPs → 16/4608 ≈ 0.35 % of peak, 16/2896 ≈ 0.55 % of HPL.

Expected result: < 2 % of peak because f(x) is scalar — no SIMD, spike branch defeats auto-vectorisation. Compute-ceiling limited, but scalar execution achieves ~1 FLOPs/cycle vs 16 FLOPs/cycle with AVX2 FMA.

### Which ceiling is more honest?

**HPL-achievable (2896 GFLOPs) is the more meaningful comparison** for non-DGEMM code. The 4608 GFLOPs theoretical peak requires fully-pipelined AVX2 FMA across all 128 cores — only vendor-tuned DGEMM approaches it. HPL is "the ceiling actually attained by carefully-tuned compute-bound code" and is the fairer benchmark for realistic kernels. See [[../performance/Roofline Model]] and [[../performance/STREAM and HPL]].

### Section 4 — What you'd try next

Good options:
- **SIMD-vectorise f(x)** using `#pragma omp simd` — the spike branch must become a blend mask. Predicted: 4–8× base-case throughput; limited on spike path.
- **Tune chunk size** — profile dynamic(32) vs dynamic(128) at 128T to find the overhead/balance crossover.
- **Increase N** to verify the kernel isn't hitting L3 cache effects at the current problem size.

## Common Mistakes

| Mistake | Effect | Fix |
|---|---|---|
| Forgetting `default(none)` | Implicit sharing of accumulator → race | Add `default(none) shared(a, h, n) reduction(+:sum)` |
| Including `0.5*(f(a)+f(b))` inside the parallel region | Double-counts endpoints | Keep endpoint terms outside, above the `#pragma` |
| Leaving `schedule(runtime)` in the final submission | Grader runs without `OMP_SCHEDULE` set → silently falls back to `static` → worst-case load imbalance at 128T → 0–1 pt performance score | After the CX3 sweep, hardcode the winner: e.g. `schedule(dynamic, 64)` |
| Not warming up before timing | First region pays runtime startup → T(1) looks 5–10× slower | Run a no-op parallel region or use `hyperfine --warmup 1` |
| Same schedule for all thread counts without measuring | Misses that static collapses at 128T | Sweep all 3 + chunk sizes; record all |
| `efficiency = speedup` in tables.csv | Fails internal-consistency check | Divide speedup by P |
| Not running on a full Rome node | Shared login node times are noisy | Use `qsub` with `select=1:ncpus=128:cpu_type=rome:place=excl` |
| Committing `build/` directory | CI rejects the push | Keep `build/` in `.gitignore` |
